# WikiDev — Documentation Overview

Este directorio contiene la documentación principal del proyecto **WikiDev**.  
Cada archivo proporciona información clave sobre la arquitectura, flujo de trabajo, alcance y ejecución del proyecto.

---

## Contenido de `docs/`

| Archivo | Descripción |
|---------|-------------|
| **ARCHITECTURE.md** | Describe la arquitectura general del proyecto, incluyendo frontend, backend, estructura de carpetas, responsabilidades de cada componente, flujo de comunicación y consideraciones de seguridad. Ideal para entender cómo está construido WikiDev y cómo interactúan sus partes. |
| **DESCRIPTION_AND_SCOPE.md** | Explica qué es WikiDev, el alcance del proyecto, objetivos, usuarios, limitaciones y supuestos. Útil para comprender qué funcionalidades se incluyen y cuáles quedan fuera del alcance. |
| **EXECUTION.md** | Documenta cómo se ejecuta el código en el proyecto, incluyendo los lenguajes soportados, flujo de ejecución, medidas de seguridad, limitaciones y ejemplos de integración en el frontend y backend. |
| **GIT-WORKFLOW.md** | Contiene la estrategia de Git, flujo de trabajo de ramas (`feat/*`, `fix/*`, `develop`, `main`, etc.), convención de commits, PR, integración continua y buenas prácticas para mantener el repositorio organizado y profesional. |
| **CHANGELOG.md** | Registro histórico de cambios y versiones del proyecto, incluyendo nuevas funcionalidades, fixes, mejoras y notas de release. Sirve como referencia para el equipo y para futuros mantenedores. |
| **TESTS.md** | (Pendiente) Documento que describirá cómo se organizan y ejecutan los tests en frontend y backend, convenciones de testing y buenas prácticas para asegurar calidad de código. |

---

## Recomendaciones de uso

- Antes de comenzar a desarrollar, leer **GIT-WORKFLOW.md** para entender las ramas y PRs.  
- Consultar **ARCHITECTURE.md** y **DESCRIPTION_AND_SCOPE.md** para conocer la estructura y alcance del proyecto.  
- Revisar **EXECUTION.md** antes de crear ejemplos de código ejecutables en MDX o notas.  
- Mantener **CHANGELOG.md** actualizado tras cada release o cambio importante.  
- Utilizar **TESTS.md** como guía para crear y ejecutar tests de manera consistente.  

---

Este README sirve como punto de partida para explorar la documentación interna y asegurar que todos los miembros del proyecto comprendan su estructura y propósito.
